基于 Claude Code 工作流设计，结合 Sub2API 项目经验

--------------------------------------------------------------------------------
一、项目概述
项目说明名称TokenViz Lite定义基于代理网关的个人 AI API Token 消耗可视化工具——零代码侵入，改一个 base_url 就能追踪所有模型的用量和成本技术栈TypeScript 全栈（Fastify + Express + Vite + React + Tremor）数据库PostgreSQL 16包管理pnpm部署Docker Compose
核心理念
传统方式（SDK 推送）：你要改代码，加 SDK，每段脚本都要初始化
代理方式（本项目） ：你改一个 base_url，其他什么都不动，代理自动记录一切

项目定位
维度说明目标用户个人开发者、独立开发者、AI 应用开发者核心价值零侵入追踪多家 LLM API 的 token 消耗和成本vs Sub2APISub2API 是 API 分发平台（管人管钱），本项目是个人消耗洞察工具（看数据做优化）vs LangfuseLangfuse 是全链路可观测平台（需要嵌入 SDK），本项目是代理网关（零代码侵入）
参考 Sub2API 的具体内容
可借鉴具体参考SSE 流式尾捕获思路backend/internal/pkg/antigravity/stream_transformer.go — SSE 逐行解析框架各厂商 usage 字段结构backend/internal/pkg/antigravity/claude_types.go L122-L129 (ClaudeUsage)Anthropic SSE 事件类型backend/internal/pkg/apicompat/types.go L126-L141 (AnthropicStreamEvent)Gemini 缓存 token 减法backend/internal/pkg/antigravity/stream_transformer.go L112-L118Usage 数据模型字段backend/ent/schema/usage_log.go L32-L146成本字段精度backend/ent/schema/usage_log.go L84-L99 — 必须用 DECIMAL 不用 FLOATTTFT 采集backend/internal/handler/openai_gateway_handler.go L728-L730

为什么 gateway 和 API 合并但分入口
- 共享类型定义和数据库连接，避免代码重复
- 部署时用同一个 Docker 镜像，entry-proxy.ts 和 entry-api.ts 启动不同进程
- docker-compose 里写两个 service 指向同一镜像，各自独立重启互不影响
- 参考 Sub2API 的做法：handler（代理）和 admin handler（查询）在同一个 Go 进程里，但路由隔离

6.1 SSE 尾捕获（streaming.ts 核心逻辑）
// 原理：逐 chunk 转发给客户端（零延迟），同时 peek 有没有 usage 字段
// 参考 Sub2API: backend/internal/pkg/antigravity/stream_transformer.go L76-L148

interface CapturedUsage {
  prompt_tokens: number;
  completion_tokens: number;
  total_tokens: number;
  cache_read_tokens: number;
  cache_creation_tokens: number;
  cache_creation_5m_tokens: number;
  cache_creation_1h_tokens: number;
}

async function handleStreamingRequest(
  upstream: Response,
  clientRes: ServerResponse,
  provider: string,
  model: string,
  startTime: number
): Promise<void> {
  // 设置 SSE 响应头
  clientRes.writeHead(200, {
    'Content-Type': 'text/event-stream',
    'Cache-Control': 'no-cache',
    'Connection': 'keep-alive',
  });

  let capturedUsage: CapturedUsage | null = null;
  let firstTokenTime: number | null = null;
  const decoder = new TextDecoder();

  for await (const chunk of upstream.body as AsyncIterable<Buffer>) {
    // 1. 立即转发（零延迟，不阻塞）
    clientRes.write(chunk);
    
    if (!firstTokenTime) firstTokenTime = Date.now();

    // 2. 顺手 peek usage（try-catch 保证解析失败不影响转发）
    const text = decoder.decode(chunk, { stream: true });
    for (const line of text.split('\n')) {
      if (!line.startsWith('data: ') || line.includes('[DONE]')) continue;
      try {
        const json = JSON.parse(line.slice(6));
        // 根据厂商提取 usage
        const usage = extractUsageByProvider(json, provider);
        if (usage) capturedUsage = { ...capturedUsage, ...usage };
      } catch { /* 解析失败无所谓，下一个 chunk 再试 */ }
    }
  }

  clientRes.end();

  // 3. 流结束，异步写库
  if (capturedUsage) {
    const latency = Date.now() - startTime;
    const ttft = firstTokenTime ? firstTokenTime - startTime : null;
    await recordUsage({ provider, model, ...capturedUsage, latency_ms: latency, first_token_ms: ttft, is_streaming: true });
  }
}

6.2 各厂商 usage 提取（providers/）
// === openai.ts — OpenAI + DeepSeek/智谱/通义 ===
// 参考: backend/internal/pkg/apicompat/types.go L113-L119 (AnthropicUsage 同理)

function extractOpenAIUsage(json: any): CapturedUsage | null {
  if (!json.usage) return null;
  const u = json.usage;
  return {
    prompt_tokens: u.prompt_tokens ?? 0,
    completion_tokens: u.completion_tokens ?? 0,
    total_tokens: u.total_tokens ?? 0,
    // OpenAI 缓存在嵌套对象里
    cache_read_tokens: u.prompt_tokens_details?.cached_tokens ?? 0,
    cache_creation_tokens: 0,  // OpenAI 无此字段
    cache_creation_5m_tokens: 0,
    cache_creation_1h_tokens: 0,
    // DeepSeek 特殊字段
    ...(u.prompt_cache_hit_tokens !== undefined && {
      cache_read_tokens: u.prompt_cache_hit_tokens,
    }),
  };
}

// === anthropic.ts — Anthropic 格式 ===
// 参考: backend/internal/pkg/apicompat/types.go L126-L141

function extractAnthropicUsage(json: any): CapturedUsage | null {
  // message_start 事件：包含 input_tokens + cache 字段
  if (json.type === 'message_start' && json.message?.usage) {
    const u = json.message.usage;
    return {
      prompt_tokens: u.input_tokens ?? 0,
      completion_tokens: 0,  // 还没有 output
      total_tokens: u.input_tokens ?? 0,
      cache_read_tokens: u.cache_read_input_tokens ?? 0,
      cache_creation_tokens: u.cache_creation_input_tokens ?? 0,
      cache_creation_5m_tokens: 0,  // Anthropic 不区分 5m/1h，由 cache_control TTL 决定
      cache_creation_1h_tokens: 0,
    };
  }
  // message_delta 事件：补充 output_tokens
  if (json.type === 'message_delta' && json.usage) {
    return {
      prompt_tokens: 0,
      completion_tokens: json.usage.output_tokens ?? 0,
      total_tokens: json.usage.output_tokens ?? 0,
      cache_read_tokens: 0,
      cache_creation_tokens: 0,
      cache_creation_5m_tokens: 0,
      cache_creation_1h_tokens: 0,
    };
  }
  return null;
}

// === gemini.ts — Gemini 格式 ===
// 关键：Gemini 的 promptTokenCount 包含 cachedContentTokenCount，要减掉
// 参考: backend/internal/pkg/antigravity/stream_transformer.go L112-L118

function extractGeminiUsage(json: any): CapturedUsage | null {
  const meta = json.usageMetadata;
  if (!meta) return null;
  const cached = meta.cachedContentTokenCount ?? 0;
  return {
    prompt_tokens: meta.promptTokenCount - cached,  // 关键减法
    completion_tokens: meta.candidatesTokenCount ?? 0,
    total_tokens: (meta.promptTokenCount ?? 0) + (meta.candidatesTokenCount ?? 0),
    cache_read_tokens: cached,
    cache_creation_tokens: 0,
    cache_creation_5m_tokens: 0,
    cache_creation_1h_tokens: 0,
  };
}

6.3 成本计算（pricing.ts）
// 从 model_pricing 表查价格，按 token 数计算成本
async function calculateCost(
  provider: string,
  model: string,
  usage: CapturedUsage
): Promise<{ input_cost: number; output_cost: number; total_cost: number }> {
  const pricing = await getPricing(provider, model);
  if (!pricing) return { input_cost: 0, output_cost: 0, total_cost: 0 };

  const MTok = 1_000_000;
  const input_cost = (usage.prompt_tokens * pricing.input_price / MTok)
                   + (usage.cache_creation_tokens * pricing.cache_write_price / MTok);
  const output_cost = (usage.completion_tokens * pricing.output_price / MTok);
  // cache_read 用 cache_read_price 计算（通常是 input 的 10%-50%）
  const cache_cost = usage.cache_read_tokens * pricing.cache_read_price / MTok;

  return {
    input_cost: input_cost + cache_cost,
    output_cost,
    total_cost: input_cost + cache_cost + output_cost,
  };
}

# TokenViz Lite — Claude Code 指令

## 项目概述
基于代理网关的个人 AI API Token 消耗可视化工具。

## 技术栈
- 后端: TypeScript + Fastify (代理) + Express (查询 API)
- 前端: Vite + React + Tremor
- 数据库: PostgreSQL 16
- 包管理: pnpm

## 关键架构决策
1. 代理和查询 API 共享一个 package，通过不同入口启动
2. SSE 流式使用"尾捕获"方案，不是逐 chunk 完整解析
3. 成本字段必须用 DECIMAL，不用 FLOAT
4. pricing.json 是定价数据的唯一真实来源，启动时导入数据库
5. usage 写库必须异步（不阻塞代理转发）


## 常用命令
```bash

cd gateway && pnpm dev:api         # 查询 API 进程

# 前端开发

# 构建
cd gateway && pnpm build
cd web && pnpm build

# 测试
cd gateway && pnpm test
cd web && pnpm test

# 数据库
psql -U tokenviz -h 127.0.0.1 -d tokenviz -f src/db/schema.sql

参考 Sub2API 的代码
- SSE 流解析框架: backend/internal/pkg/antigravity/stream_transformer.go
- 各厂商类型定义: backend/internal/pkg/apicompat/types.go
- Usage 数据模型: backend/ent/schema/usage_log.go
- Gemini 缓存 token 减法: stream_transformer.go L112-L118
- TTFT 采集: openai_gateway_handler.go L728-L730
常见坑点
1.DECIMAL 精度：成本字段不能用 FLOAT，浮点累加会出问题
2.Gemini 缓存 token：promptTokenCount 包含 cachedContentTokenCount，必须减掉

4.SSE 解析失败：try-catch 包裹，解析失败不影响转发
5.连接中断：流式请求可能收不到 usage，记录一条 usage=null 的日志
6.pnpm-lock.yaml：改了 package.json 必须同步提交 lock 文件

## 十一、风险与缓解

| 风险 | 影响 | 缓解 |
|------|------|------|
| SSE 格式变更 | usage 提取失败 | try-catch 保证不影响转发；记录原始 chunk 到 debug 日志 |
| 新模型无定价 | 成本计算为 0 | 使用默认定价（按 input/output 各 $1/MTok），仪表板标注"估算" |
| 数据库写入延迟 | 聚合数据不实时 | usage_logs 表可实时查询，聚合表只是加速 |
| 代理进程挂掉 | 所有请求中断 | docker 自动重启；v2 考虑多实例 |
| 定价数据过时 | 成本不准确 | 仪表板显示定价最后更新时间；提供一键更新入口 |
